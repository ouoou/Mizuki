# Java并发知识点面试总结（口语化版本）

## 一、基础概念：并发与并行、进程与线程

### 1. 并发和并行的区别

面试官可能会问："你能说说并发和并行有什么区别吗？"

**口语化回答：**

并发和并行这两个概念经常被混淆，但其实它们有本质区别。

**并发（Concurrency）** 指的是在一段时间内，多个任务看起来是同时进行的，但实际上可能是通过时间片轮转的方式，在一个CPU核心上交替执行。就像你一边听音乐一边写代码，看起来是同时做的，但实际上CPU在快速切换，一会儿处理音乐播放，一会儿处理代码编辑。

**并行（Parallelism）** 则是真正意义上的同时执行，多个任务在多个CPU核心上真正同时运行。比如一个8核CPU，可以同时运行8个任务，这就是并行。

在Java中，我们通常说的多线程编程，大多数情况下是并发，因为线程可能运行在同一个CPU核心上，通过上下文切换来实现。但如果你的程序运行在多核CPU上，并且线程数不超过核心数，那就可以实现真正的并行。

**应用场景：**
- 并发更适合I/O密集型任务，比如网络请求、文件读写，因为等待I/O的时候CPU可以切换到其他任务
- 并行更适合CPU密集型任务，比如图像处理、科学计算，需要真正的多核计算能力

### 2. 进程和线程的关系

**口语化回答：**

进程是操作系统分配资源的基本单位，每个进程都有自己独立的内存空间，进程之间是隔离的，一个进程崩溃不会影响其他进程。线程是CPU调度的基本单位，一个进程可以包含多个线程，这些线程共享进程的内存空间，但每个线程有自己的栈空间和程序计数器。

在Java中，JVM本身就是一个进程。当我们创建Java线程时，这些线程通常与操作系统的线程是一对一映射的，也就是说一个Java线程对应一个OS线程。所有Java线程共享JVM的堆内存，但每个线程有自己的栈空间，用来存储局部变量、方法调用等信息。

为什么需要线程？因为线程比进程更轻量级，创建和切换的开销更小。而且线程之间共享内存，通信更方便。但这也带来了线程安全问题，因为多个线程可以同时访问共享数据。

### 3. 上下文切换

**口语化回答：**

上下文切换是CPU从执行一个线程切换到执行另一个线程的过程。这个过程需要保存当前线程的状态（比如寄存器值、程序计数器、栈指针等），然后加载下一个线程的状态。

上下文切换什么时候发生？主要有几种情况：
1. 时间片用完了，操作系统强制切换
2. 线程主动让出CPU，比如调用sleep、wait、yield
3. 线程被阻塞了，比如等待I/O、等待锁
4. 更高优先级的线程需要运行

上下文切换是有开销的，因为需要保存和恢复状态，还会导致CPU缓存失效。所以如果上下文切换太频繁，会影响性能。

**优化策略：**
- 合理设置线程池大小，避免创建过多线程
- 减少锁竞争，使用无锁数据结构
- 对于I/O密集型任务，可以考虑使用虚拟线程（JDK 21的新特性）

## 二、Java内存模型（JMM）与硬件基础

### 4. 什么是Java内存模型（JMM）

**口语化回答：**

Java内存模型（JMM）是Java虚拟机定义的一个抽象规范，它规定了多线程环境下，线程如何与内存交互，以及如何保证内存的可见性、有序性。

JMM把内存分为主内存和工作内存。主内存是所有线程共享的，工作内存是每个线程私有的。线程对变量的操作都在工作内存中进行，然后同步到主内存。这其实是对CPU缓存的一个抽象，因为现代CPU都有多级缓存，每个核心有自己的缓存，这就对应了工作内存的概念。

JMM的核心是happens-before规则，它定义了哪些操作必须发生在哪些操作之前，从而保证可见性和有序性。主要有这么几条：
1. **程序顺序规则**：单线程内，代码顺序执行
2. **volatile规则**：volatile变量的写操作happens-before后续的读操作
3. **监视器锁规则**：解锁操作happens-before后续的加锁操作
4. **线程启动规则**：Thread.start() happens-before线程内的任何操作
5. **线程终止规则**：线程内的所有操作happens-before线程终止
6. **传递性**：如果A happens-before B，B happens-before C，那么A happens-before C

### 5. 有了MESI为什么还需要JMM

**口语化回答：**

这是一个很好的问题。MESI是CPU缓存一致性协议，它保证了多个CPU核心的缓存之间数据的一致性。但MESI只解决了硬件层面的缓存一致性问题，它解决不了以下几个问题：

1. **指令重排序**：编译器和CPU为了优化性能，可能会重排序指令。MESI只保证缓存一致性，不保证指令顺序。
2. **平台差异**：不同的CPU架构（x86、ARM）有不同的内存模型，Java需要一套统一的规范。
3. **语言层面的语义**：Java需要定义volatile、synchronized这些关键字的具体语义，MESI只是底层实现的一部分。

所以JMM是在MESI等硬件机制之上，提供了一套语言层面的内存模型规范，让Java程序在不同平台上都能有正确的并发语义。

### 6. 总线嗅探和总线风暴

**口语化回答：**

总线嗅探是CPU缓存一致性的一种实现方式。当某个CPU核心修改了缓存中的数据时，会通过系统总线发送消息，其他CPU核心会"嗅探"这个总线，如果发现自己的缓存中有这个数据的副本，就会更新缓存状态（比如从Shared变成Invalid）。

总线风暴是指在高并发场景下，大量的缓存失效操作导致系统总线被占满，成为性能瓶颈。这通常发生在：
1. 大量线程同时竞争同一个volatile变量
2. 频繁的synchronized操作导致缓存行失效
3. 伪共享（False Sharing）问题，多个变量在同一个缓存行上

**优化方法：**
- 避免伪共享，可以使用`@Contended`注解或者手动填充
- 对于高并发的计数器，使用`LongAdder`而不是`AtomicLong`
- 减少不必要的volatile和synchronized使用

## 三、线程状态与线程安全

### 7. 线程的几种状态

**口语化回答：**

Java中线程有6种状态，定义在`Thread.State`枚举中：

1. **NEW（新建）**：线程对象创建了，但还没调用start()
2. **RUNNABLE（可运行）**：这个状态包括两种情况：就绪（Ready）和运行中（Running）。Java没有单独区分这两种状态，因为这是操作系统层面的概念
3. **BLOCKED（阻塞）**：线程在等待获取一个监视器锁，比如进入synchronized代码块时，锁被其他线程持有
4. **WAITING（等待）**：无限期等待，需要其他线程唤醒。比如调用Object.wait()、Thread.join()、LockSupport.park()
5. **TIMED_WAITING（超时等待）**：有超时时间的等待，比如Thread.sleep()、Object.wait(timeout)
6. **TERMINATED（终止）**：线程执行完毕

**状态转换：**
- NEW → RUNNABLE：调用start()
- RUNNABLE → BLOCKED：尝试获取锁失败
- RUNNABLE → WAITING：调用wait()、join()、park()
- RUNNABLE → TIMED_WAITING：调用sleep()、wait(timeout)
- 各种等待状态 → RUNNABLE：被唤醒或超时
- RUNNABLE → TERMINATED：线程执行完毕

### 8. 什么是线程安全

**口语化回答：**

线程安全是指多线程环境下，代码能够正确执行，不会出现数据不一致、脏读、丢失更新等问题。

线程安全需要保证三个特性：
1. **原子性（Atomicity）**：操作要么全部执行，要么都不执行，不会被其他线程打断。比如i++这个操作，实际上包含读、改、写三步，不是原子的。
2. **可见性（Visibility）**：一个线程修改了共享变量，其他线程能立即看到这个修改。
3. **有序性（Ordering）**：代码的执行顺序符合预期，不会因为指令重排序导致逻辑错误。

**为什么会有线程安全问题？**
主要原因是：
1. **CPU缓存**：每个CPU核心有自己的缓存，修改可能只写入了缓存，没有立即同步到主内存
2. **指令重排序**：编译器和CPU为了优化，可能改变指令执行顺序

**解决方案：**
- `synchronized`：保证原子性、可见性、有序性
- `volatile`：保证可见性和有序性，但不保证原子性
- `Atomic`类：使用CAS保证原子性
- 不可变对象：如果对象不可变，就不存在线程安全问题
- 线程安全的集合类：比如`ConcurrentHashMap`

## 四、volatile关键字

### 9. volatile的实现原理

**口语化回答：**

volatile是Java提供的一个轻量级同步机制，它有两个作用：
1. **保证可见性**：一个线程修改了volatile变量，其他线程能立即看到
2. **禁止指令重排序**：volatile变量前后的指令不会被重排序

**底层实现：**
volatile通过内存屏障（Memory Barrier）来实现。内存屏障是一组CPU指令，用来禁止特定类型的指令重排序，并保证内存可见性。

主要有四种内存屏障：
- **LoadLoad**：确保Load1在Load2之前执行
- **StoreStore**：确保Store1在Store2之前执行
- **LoadStore**：确保Load在Store之前执行
- **StoreLoad**：确保Store在Load之前执行（这是最重的屏障，volatile写后会插入这个）

在x86架构上，StoreLoad屏障通常对应`mfence`指令。JIT编译器会在volatile变量的读写前后插入相应的内存屏障。

**与MESI协议的关系：**
当写入volatile变量时，由于内存屏障的作用，会强制将缓存行写回主内存，并让其他CPU核心的缓存失效。MESI协议会处理这个失效过程，保证其他核心读取时能拿到最新值。

### 10. volatile能保证原子性吗？

**口语化回答：**

**不能！** 这是volatile最容易让人误解的地方。

volatile只能保证可见性和有序性，不能保证原子性。比如`i++`这个操作，虽然i是volatile的，但`i++`实际上包含三个步骤：
1. 读取i的值
2. 将i加1
3. 写回i

这三个步骤不是原子的，如果两个线程同时执行，可能会出现：
- 线程A读取i=1
- 线程B读取i=1
- 线程A计算i+1=2，写回
- 线程B计算i+1=2，写回
- 结果i=2，但应该是3

**volatile适合什么场景？**
- 简单的标志位，比如`volatile boolean flag`
- 单写多读的场景，只有一个线程写，多个线程读
- 作为其他同步机制的辅助，比如double-checked locking中的instance变量

如果需要原子性，应该使用：
- `synchronized`
- `AtomicInteger`等原子类
- `ReentrantLock`

## 五、synchronized关键字

### 11. synchronized是怎么实现的

**口语化回答：**

synchronized是Java内置的同步机制，它是一个JVM层面的监视器锁（Monitor Lock）。

**字节码层面：**
当我们使用synchronized时，编译器会在字节码中插入`monitorenter`和`monitorexit`指令。`monitorenter`尝试获取锁，`monitorexit`释放锁。

**对象头（Object Header）：**
每个Java对象都有一个对象头，其中包含Mark Word，用来存储对象的哈希码、GC年龄、锁状态等信息。synchronized就是通过修改Mark Word来实现锁的。

**锁的升级过程：**
synchronized的锁不是一开始就是重量级锁，而是有一个升级过程：

1. **无锁状态**：对象刚创建时
2. **偏向锁（Biased Lock）**：如果只有一个线程访问，JVM会启用偏向锁，在Mark Word中记录线程ID。这样后续这个线程访问时，不需要CAS，直接检查线程ID即可。偏向锁在JDK 15后被废弃了，因为现代应用多线程竞争激烈，偏向锁的收益不大，反而撤销开销大。
3. **轻量级锁（Lightweight Lock）**：当有线程竞争时，升级为轻量级锁。线程会在自己的栈中创建一个Lock Record，然后通过CAS尝试将Mark Word指向这个Lock Record。如果CAS成功，就获取了锁；如果失败，说明有竞争，会自旋几次尝试获取。
4. **重量级锁（Heavyweight Lock）**：如果轻量级锁竞争激烈，自旋失败，就会升级为重量级锁。这时会创建一个ObjectMonitor对象，线程会被阻塞，进入等待队列，需要操作系统来调度唤醒。

**为什么JDK 15要废弃偏向锁？**
1. 现代应用多线程竞争激烈，偏向锁很少能发挥作用
2. 偏向锁的撤销需要进入安全点（Safepoint），开销大
3. 简化JVM实现，减少代码复杂度

**synchronized的特性：**
- **可重入**：同一个线程可以多次获取同一个锁
- **非公平锁**：不保证等待时间最长的线程先获取锁
- **自动释放**：即使发生异常，锁也会自动释放

### 12. synchronized的优化

**口语化回答：**

JVM对synchronized做了很多优化：

1. **锁消除（Lock Elimination）**：如果JVM检测到某个锁不可能被其他线程访问，就会消除这个锁。比如局部变量，不可能被其他线程访问，就不需要加锁。

2. **锁粗化（Lock Coarsening）**：如果一段代码中有连续的多个synchronized块，JVM可能会将它们合并成一个，减少锁的获取和释放次数。

3. **自适应自旋（Adaptive Spinning）**：在轻量级锁升级为重量级锁之前，线程会自旋尝试获取锁。自旋次数不是固定的，而是根据之前的成功率动态调整。如果之前自旋成功率高，就多自旋几次；如果成功率低，就少自旋或直接阻塞。

4. **偏向锁**：虽然被废弃了，但在JDK 15之前，它确实减少了无竞争情况下的开销。

## 六、CAS（Compare-And-Swap）

### 13. CAS的实现原理

**口语化回答：**

CAS（Compare-And-Swap）是一种无锁编程技术，它是一个原子操作，包含三个参数：内存地址V、期望值A、新值B。只有当V的值等于A时，才会将V更新为B，否则不做任何操作。无论是否更新，都会返回V的原始值。

**硬件实现：**
CAS是CPU提供的原子指令：
- **x86架构**：`LOCK CMPXCHG`指令，LOCK前缀保证原子性
- **ARM架构**：`LL/SC`（Load-Link/Store-Conditional）指令对

**CAS的优点：**
- 无锁，避免了线程阻塞和上下文切换
- 适合低竞争场景，性能好

**CAS的问题：**
1. **ABA问题**：值从A变成B又变回A，CAS会认为没有变化。可以用`AtomicStampedReference`解决，通过版本号来避免ABA问题。
2. **自旋开销**：如果竞争激烈，线程会一直自旋，浪费CPU
3. **只能保证一个变量的原子性**：如果需要保证多个变量的原子性，还是需要锁
4. **总线风暴**：高并发下大量CAS操作会导致缓存失效，总线被占满

**为什么CAS需要volatile？**
CAS操作的是内存地址，但如果这个地址对应的变量不是volatile的，其他线程可能看不到最新的值，导致CAS基于旧值进行比较，出现错误。

### 14. 高并发下CAS与锁的选择

**口语化回答：**

这是一个性能优化的关键问题。

**CAS在高并发下的问题：**
1. **CPU自旋浪费**：大量线程同时CAS，大部分会失败，但还在不断重试，浪费CPU
2. **总线风暴**：频繁的缓存失效导致系统总线饱和
3. **伪共享**：多个变量在同一个缓存行，一个变量更新导致整个缓存行失效

**锁在高并发下的优势：**
1. **阻塞而非自旋**：线程获取不到锁会被阻塞，不浪费CPU
2. **有序等待**：通过等待队列，保证公平性
3. **减少总线压力**：只有获取锁的线程在修改，减少了缓存失效

**选择策略：**
- **低竞争场景**：CAS性能更好，因为无锁，没有上下文切换
- **高竞争场景**：锁（特别是synchronized的锁升级机制）性能更好，因为避免了无意义的自旋
- **中等竞争**：可以使用自适应策略，比如AQS的实现，先自旋几次，失败后再阻塞

synchronized的锁升级机制实际上就是一种自适应策略：无竞争用偏向锁，低竞争用轻量级锁+自旋，高竞争用重量级锁+阻塞。

## 七、AQS（AbstractQueuedSynchronizer）

### 15. AQS的实现原理

**口语化回答：**

AQS是Java并发包的基础框架，很多并发工具类都是基于它实现的，比如ReentrantLock、Semaphore、CountDownLatch。

**核心思想：**
AQS使用一个`volatile int state`来表示同步状态，以及一个FIFO队列（CLH队列的变种）来管理等待的线程。

**两种模式：**
1. **独占模式（Exclusive）**：同一时刻只有一个线程能获取锁，比如ReentrantLock
2. **共享模式（Shared）**：同一时刻可以有多个线程获取，比如Semaphore、CountDownLatch

**CLH队列：**
CLH队列是一个双向链表，每个节点代表一个等待的线程。节点有两种类型：
- **同步队列（Sync Queue）**：等待获取锁的线程
- **条件队列（Condition Queue）**：等待某个条件的线程（通过Condition对象）

**工作流程：**
1. 线程尝试获取锁，调用`tryAcquire()`（子类实现）
2. 如果成功，设置state，当前线程获取锁
3. 如果失败，将线程包装成节点，加入队列，然后调用`LockSupport.park()`阻塞
4. 当锁释放时，唤醒队列中的下一个线程

**为什么用CLH队列？**
CLH队列是自旋锁的一种实现，但AQS中节点是阻塞的，不是自旋的。使用CLH队列的好处是：
- FIFO，保证公平性
- 节点只需要关注前驱节点的状态，减少CAS竞争
- 支持取消和超时

### 16. 基于AQS的并发工具类

**口语化回答：**

**ReentrantLock：**
- 可重入的独占锁
- 支持公平锁和非公平锁
- 基于AQS的独占模式实现
- 相比synchronized，提供了更灵活的功能，比如可中断、可超时、支持多个条件变量

**Semaphore：**
- 信号量，控制同时访问某个资源的线程数
- 基于AQS的共享模式实现
- state表示可用许可数，acquire()获取许可，release()释放许可

**CountDownLatch：**
- 倒计时门闩，一个或多个线程等待其他线程完成
- 基于AQS的共享模式实现
- state表示计数器，countDown()减1，await()等待state变为0
- 只能使用一次，不能重置

**CyclicBarrier：**
- 循环屏障，多个线程互相等待，到达屏障点后一起继续
- 不是基于AQS实现的，而是用ReentrantLock + Condition实现的
- 可以重复使用，这是它与CountDownLatch的主要区别

**选择建议：**
- 基本同步需求用synchronized，简单直接
- 需要高级功能（可中断、公平锁、多个条件）用ReentrantLock
- 控制并发数用Semaphore
- 等待其他线程完成用CountDownLatch
- 多个线程互相等待用CyclicBarrier

## 八、ThreadLocal

### 17. ThreadLocal的实现原理

**口语化回答：**

ThreadLocal提供了线程本地存储，每个线程都有自己独立的变量副本，互不干扰。

**实现机制：**
每个Thread对象内部都有一个ThreadLocalMap，这是一个自定义的HashMap。ThreadLocal作为key，线程的变量值作为value。

**关键点：**
- ThreadLocalMap使用**弱引用**作为key（ThreadLocal对象）
- 但value是**强引用**
- 这样可以避免ThreadLocal对象无法被GC，但如果ThreadLocal被回收了，key会变成null，但value还在

**使用场景：**
- 存储用户信息、事务上下文等需要线程隔离的数据
- 避免参数传递，简化代码
- 线程安全的工具类，比如SimpleDateFormat，每个线程有自己的实例

### 18. ThreadLocal的内存泄漏问题

**口语化回答：**

这是ThreadLocal的一个经典问题。

**问题原因：**
ThreadLocalMap的key是弱引用，如果ThreadLocal对象没有其他强引用，就会被GC回收，key变成null。但value是强引用，还留在Map中。如果线程是线程池中的线程，会一直存活，这些value就永远不会被回收，造成内存泄漏。

**解决方案：**
1. **使用完及时remove()**：在finally块中调用`ThreadLocal.remove()`，清理当前线程的ThreadLocalMap中的entry
2. **使用JDK 21的ScopedValue**：这是ThreadLocal的替代方案，提供了自动清理机制，更适合虚拟线程

**最佳实践：**
```java
ThreadLocal<String> threadLocal = new ThreadLocal<>();
try {
    threadLocal.set("value");
    // 使用threadLocal
} finally {
    threadLocal.remove(); // 必须清理
}
```

## 九、虚拟线程（JDK 21）

### 19. 虚拟线程的实现原理

**口语化回答：**

虚拟线程是JDK 21引入的新特性（Project Loom），它是轻量级的、由JVM管理的线程，不是1:1映射到OS线程的。

**为什么需要虚拟线程？**
传统的"一个请求一个线程"模型在高并发I/O场景下有问题：
- 创建OS线程开销大
- 线程数量受限于OS限制
- 大量线程等待I/O时，占用内存和调度资源

虚拟线程就是为了解决这个问题，可以创建百万级别的虚拟线程，而底层只需要少量OS线程（载体线程）。

**核心组件：**
1. **Continuation（延续）**：可以暂停和恢复的执行上下文，虚拟线程的挂起和恢复就是通过Continuation实现的
2. **Carrier Thread（载体线程）**：执行虚拟线程的OS线程，一个载体线程可以执行多个虚拟线程
3. **Stack Chunking（栈分块）**：虚拟线程的栈帧可以保存到堆上，需要时再恢复，这样虚拟线程占用的内存更少

**使用方式：**
```java
// 创建虚拟线程
Thread virtualThread = Thread.ofVirtual().start(() -> {
    // 任务代码
});

// 使用虚拟线程池
ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor();
```

**限制（Pinning）：**
当虚拟线程执行以下操作时，会被"固定"到载体线程上，无法被挂起：
- 进入synchronized代码块
- 调用native方法

所以建议使用ReentrantLock而不是synchronized，避免pinning。

## 十、线程池

### 20. 线程池的实现原理

**口语化回答：**

线程池是管理线程的容器，可以复用线程，减少创建和销毁的开销。

**核心参数（ThreadPoolExecutor）：**
1. **corePoolSize**：核心线程数，即使空闲也会保留
2. **maximumPoolSize**：最大线程数
3. **keepAliveTime**：非核心线程的空闲存活时间
4. **workQueue**：任务队列，存储等待执行的任务
5. **ThreadFactory**：创建线程的工厂
6. **RejectedExecutionHandler**：拒绝策略，当线程池满了且队列也满了时的处理方式

**执行流程：**
1. 提交任务
2. 如果当前线程数 < corePoolSize，创建新线程执行
3. 如果 >= corePoolSize，将任务放入队列
4. 如果队列满了且线程数 < maximumPoolSize，创建新线程执行
5. 如果线程数达到maximumPoolSize且队列满了，执行拒绝策略

**拒绝策略：**
- AbortPolicy：直接抛异常
- CallerRunsPolicy：由调用线程执行
- DiscardPolicy：直接丢弃
- DiscardOldestPolicy：丢弃最老的任务

**关闭线程池：**
- `shutdown()`：不再接受新任务，等待已有任务完成
- `shutdownNow()`：尝试停止所有任务，返回未执行的任务列表

## 十一、原子性与数据库ACID

### 21. 并发编程中的原子性和数据库ACID的原子性

**口语化回答：**

虽然都叫"原子性"，但它们的含义完全不同。

**并发编程中的原子性：**
指的是一个操作或一系列操作是不可分割的，要么全部执行，要么都不执行，不会被其他线程打断。比如`AtomicInteger.incrementAndGet()`是原子的，但`i++`不是。

**数据库ACID中的原子性：**
指的是一个事务（Transaction）中的所有操作要么全部成功，要么全部失败回滚。比如转账操作，扣款和加款必须同时成功或同时失败。

**主要区别：**
1. **范围不同**：并发原子性针对单个操作，数据库原子性针对整个事务（可能包含多个SQL）
2. **失败处理**：并发原子性如果失败，需要手动重试；数据库原子性会自动回滚
3. **实现方式**：并发原子性用CAS或锁；数据库原子性用日志（WAL）和两阶段提交（2PC）

**它们的关系：**
在实际应用中，两者经常需要配合使用。比如分布式事务，既需要保证单个操作的原子性（并发控制），也需要保证整个事务的原子性（分布式一致性）。

## 十二、知识点串联总结

### 完整的知识体系

**口语化总结：**

Java并发编程是一个完整的知识体系，从底层硬件到上层应用，层层递进。

**底层基础：**
现代CPU有多级缓存，每个核心有自己的缓存，这就带来了缓存一致性问题。MESI协议通过总线嗅探保证了缓存一致性，但只解决了硬件层面的问题。

**JMM的作用：**
Java内存模型（JMM）在硬件之上，定义了Java程序的内存语义。它通过happens-before规则，保证了可见性、有序性，并抽象了不同硬件平台的差异。volatile和synchronized都是基于JMM实现的。

**同步机制：**
- **volatile**：轻量级，保证可见性和有序性，通过内存屏障实现，适合简单的标志位
- **synchronized**：重量级，保证原子性、可见性、有序性，通过对象头Mark Word和锁升级实现，适合大多数同步场景
- **CAS**：无锁编程，适合低竞争场景，但高竞争下会有总线风暴问题

**AQS框架：**
AQS是并发工具类的基础，它用volatile state和CLH队列管理线程。ReentrantLock、Semaphore、CountDownLatch都是基于AQS实现的，提供了比synchronized更灵活的功能。

**线程管理：**
- **ThreadLocal**：线程本地存储，但要注意内存泄漏，使用完要remove()
- **线程池**：复用线程，减少开销，需要合理配置参数
- **虚拟线程**：JDK 21的新特性，适合I/O密集型高并发场景

**设计原则：**
1. **能不用锁就不用锁**：优先考虑无锁数据结构、不可变对象
2. **能用轻量级就不用重量级**：volatile能满足就用volatile，synchronized能满足就不用ReentrantLock
3. **根据场景选择**：低竞争用CAS，高竞争用锁；I/O密集型考虑虚拟线程，CPU密集型用线程池
4. **理解底层原理**：知道volatile、synchronized、CAS的底层实现，才能做出正确的选择

**常见问题：**
- 线程安全问题：原子性、可见性、有序性
- 性能问题：上下文切换、总线风暴、锁竞争
- 内存问题：ThreadLocal泄漏、线程池配置不当

掌握这些知识点，就能在面试中游刃有余地讨论Java并发编程了。
