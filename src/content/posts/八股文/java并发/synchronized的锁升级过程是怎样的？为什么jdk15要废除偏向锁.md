## What - 是什么

### 🎯 核心定义
**synchronized 锁升级是 JVM 根据竞争程度在偏向锁→轻量级锁→重量级锁间渐进升级的过程；JDK15 起默认移除偏向锁，简化为轻量/重量两级。**

### 🔍 本质特征
- **对象头驱动**: Mark Word 记录锁标志与线程信息。
- **渐进式升级**: 先偏向、再轻量自旋、最后重量级阻塞。
- **只升不降**: 降级成本高，JVM 不做自动降级。

### 📐 基本结构/组成（如适用）
```
Mark Word 锁标志位: 01(无锁/偏向标志) -> 00(轻量) -> 10(重量/Monitor)
线程栈 Lock Record: 保存旧 Mark Word，CAS 绑定对象
ObjectMonitor: owner/EntryList/WaitSet
```

### 📝 关键术语
| 术语   | 说明                               |
| ---- | -------------------------------- |
| 偏向锁  | 将线程ID写入对象头，单线程复入无CAS             |
| 轻量级锁 | CAS 绑定 Lock Record，自旋等待          |
| 重量级锁 | Mark Word 指向 ObjectMonitor，阻塞/唤醒 |
| 撤销偏向 | 竞争出现时撤销偏向，进入轻量流程                 |

## Why - 为什么

### 🧠 设计原理
**以分层锁策略在不同竞争强度下权衡开销：无竞争时极低成本，轻度竞争用自旋，重度竞争用阻塞。**

### 💡 解决的问题
- **降低无竞争成本**: 偏向锁去掉 CAS → **解决方案**: 直接认主线程。
- **降低轻度竞争上下文切换**: 轻量级自旋 → **解决方案**: 短暂忙等不入内核。
- **处理重度竞争**: 重量级锁阻塞 → **解决方案**: 交由内核调度。

### ⚡ 优势与局限
**优势**:
- **渐进成本**: 根据竞争强度匹配不同代价。
- **语法简单**: 语言级关键字，异常安全。

**局限性**:
- **升级不可逆**: 升级后的开销无法自动回退。
- **偏向撤销成本**: 竞争频繁时偏向撤销带额外停顿。
- **公平性缺失**: 默认非公平，存在饥饿风险。

## When - 什么时候用

### ✅ 适用场景
1. **低到中等竞争的互斥**: 短临界区的计数、缓存更新。
2. **无需超时/公平/条件队列**: 简单互斥即可。

### ❌ 不适用场景
- **高竞争或长临界区**: 容易快速膨胀为重量级，阻塞多。
- **需要可中断/超时**: 用 ReentrantLock 更合适。

### 🎯 判断标准
> **什么时候选择这个方案？**
> - 竞争可控且临界区短
> - 不需要锁获取超时/公平/多条件队列

### 📊 方案对比（如适用）
| 方案 | 适用场景 | 优缺点 | 选择建议 |
|------|---------|--------|----------|
| ReentrantLock | 可中断/超时/公平/条件 | 灵活但需手动解锁 | 高级同步需求 |
| synchronized | 简单互斥，锁分层优化 | 特性固定，公平性弱 | 轻量互斥首选 |

## How - 如何使用

### 🔧 基本步骤
1. **确定锁对象**: 使用固定、私有、final 的锁。
2. **包裹临界区**: 使用 synchronized 代码块或方法。
3. **缩短持锁时间**: 将非共享操作移出临界区。

### 💻 代码实现
```java
class UpgradeDemo {
  private final Object lock = new Object();
  void work() {
    synchronized (lock) { /* 临界区，可能经历偏向/轻量/重量 */ }
  }
}
```

### 📈 复杂度分析（如适用）
- **时间复杂度**: 取决于临界区逻辑；竞争时包含自旋或阻塞切换成本。
- **空间复杂度**: Mark Word + 可能的 Lock Record/ObjectMonitor。

## 底层实现原理

> **⚠️ 重要：对于需要深入理解的知识点，必须说明底层实现机制**

### 🔧 实现机制
- 字节码生成 `monitorenter/monitorexit`，JIT 进行锁消除/粗化/自旋优化。
- Mark Word 存放锁标志与线程信息，CAS/屏障驱动状态变化。

### 🏗️ 架构设计
```
Mark Word: 锁标志(偏/轻/重) + 线程ID/指针
Lock Record: 线程栈保存旧 Mark Word
ObjectMonitor: owner/EntryList/WaitSet
```

### 💾 数据结构（如适用）
- **Mark Word**: 锁标志 + 线程ID/指向锁结构。
- **Lock Record**: 轻量级锁绑定与解锁备份。
- **ObjectMonitor**: 重量级锁的等待与竞争队列。

### ⚙️ 关键机制
- **偏向锁**: 写入线程ID，无 CAS；竞争触发撤销。
- **轻量级锁**: CAS 绑定 Lock Record，自旋等待。
- **重量级锁**: Mark Word 指向 ObjectMonitor，阻塞/唤醒。
- **锁消除/粗化**: 逃逸分析去掉无效锁，合并连续锁操作。
- **自适应自旋**: 根据历史竞争调整自旋次数。

### 📝 源码分析（简化示意）
```cpp
// 偏向撤销（简化）
if (obj->mark().is_biased()) {
  revoke_bias(obj); // 停顿竞争线程，恢复为无锁或轻量锁入口
}
```

### 🔗 与底层系统的关系
- **CPU 内存屏障**: monitorenter/exit 前后发射屏障，保障可见性与有序性。
- **OS 调度**: 重量级锁的阻塞/唤醒依赖内核。
- **JIT 优化**: 锁消除/粗化、自适应自旋由 C2 负责。

### 📊 性能考虑
- 低竞争：偏向/轻量成本极低。
- 竞争上升：自旋失败膨胀，进入阻塞，开销显著。
- 偏向撤销：存在全局批量撤销与 safepoint 停顿。

## 重点疑问完整解析

### ❓ 疑问1: 锁升级的具体流程是什么？
**疑问描述**: 如何从偏向到轻量再到重量，触发条件是什么？

**完整解答**:
1. **核心答案**: 单线程复入 → 偏向锁；出现第二线程竞争撤销偏向 → 轻量级 CAS + 自旋；自旋失败或竞争激烈 → 膨胀为重量级阻塞。
2. **详细解释**: 
   - 偏向锁：Mark Word 记录线程ID，进入无需 CAS，竞争触发撤销。
   - 轻量级锁：线程在栈上创建 Lock Record，CAS 尝试把 Mark Word 设为指向 Lock Record；失败则自旋。
   - 重量级锁：自旋失败/膨胀后 Mark Word 指向 ObjectMonitor，线程进入 EntryList/WaitSet。
3. **示例说明**:
   ```text
   单线程多次进入 -> 偏向
   另一线程尝试获取 -> 撤销偏向 -> 轻量自旋
   自旋仍失败/竞争多 -> 膨胀重量级，阻塞等待
   ```
4. **常见误解**: 
   - ❌ 锁会自动降级 → ✅ 只升不降。
5. **记忆要点**: “偏向贴线程，轻量做CAS，自旋不成请内核”。

---

### ❓ 疑问2: 为什么 JDK15 要移除偏向锁？
**疑问描述**: 偏向锁曾用于低竞争场景，为何被废除？

**完整解答**:
1. **核心答案**: 偏向锁在现代应用中收益下降且撤销/维护成本高，JDK15 默认移除以简化实现、降低维护与停顿风险。
2. **详细解释**: 
   - **收益变小**: 现代应用启动快、线程多，偏向锁收益有限；C2 优化、逃逸分析已覆盖部分场景。
   - **撤销成本**: 竞争或批量撤销需要 safepoint，增加停顿与复杂度。
   - **实现维护成本**: 需要额外标志、批量撤销逻辑和配置开关。
   - **一致性**: 去除偏向后行为更稳定，可预测性更好。
3. **示例说明**:
   ```text
   JDK15: -XX:-UseBiasedLocking 默认生效，直接从无锁进入轻量级流程
   ```
4. **常见误解**: 
   - ❌ 移除偏向会让所有锁变慢 → ✅ 低竞争下轻量锁成本接近，整体影响很小。
5. **记忆要点**: “收益变小，撤销昂贵，维护复杂，干脆去掉”。

---

### ❓ 疑问3: 移除偏向锁后性能会怎样？
**疑问描述**: 无偏向时，低竞争场景性能是否下降？

**完整解答**:
1. **核心答案**: 轻量级锁在低竞争下成本接近偏向，实际差距很小；避免偏向撤销带来的停顿反而更稳。
2. **详细解释**: 
   - 轻量级锁仍在用户态 CAS + 自旋，竞争低时极快。
   - 省去批量撤销与 safepoint，减少长尾停顿。
   - JIT 还能做锁消除/粗化，覆盖多数轻量场景。
3. **示例说明**:
   ```text
   无偏向 -> 直接轻量级 CAS，自旋成功即进入
   ```
4. **常见误解**: 
   - ❌ 必然性能回退 → ✅ 多数基准显示差异微小或更稳。
5. **记忆要点**: “轻量顶上，长尾更稳”。

## 核心要点

### 🔑 核心思想
> **分层锁按竞争升级，偏向锁在 JDK15 被移除以降低撤销成本与复杂度。**

### 🎯 关键技巧
1. **锁对象固定且私有**: 避免锁失效。
2. **缩短临界区**: 减少膨胀概率与自旋时间。

### ⚠️ 易错点
| 易错点    | 错误示例       | 正确做法         | 原因       |
| ------ | ---------- | ------------ | -------- |
| 期望锁会降级 | 假设竞争结束自动降级 | 重新创建锁或复用无锁路径 | JVM 不做降级 |
| 误判偏向收益 | 高竞争仍启用偏向   | 了解版本开关，必要时关闭 | 撤销成本高    |

## 性能优化

> **⚠️ 重要：说明如何优化性能，以及性能相关的注意事项**

### ⚡ 性能特点
- **低竞争**: 轻量锁成本低。
- **高竞争**: 膨胀为重量级，阻塞与切换昂贵。
- **偏向撤销**: 在旧版本中会带来停顿成本。

### 🚀 优化策略
1. **减少共享与锁粒度**: 
   - **方法**: 拆分锁、局部变量、不可变对象。
   - **效果**: 降低竞争、减少膨胀。
   - **注意事项**: 避免锁过多导致死锁复杂度。

2. **利用锁优化与分析工具**: 
   - **方法**: 逃逸分析触发锁消除；观察自旋与阻塞。
   - **效果**: 降低 monitorenter/exit 次数，减少长尾。
   - **注意事项**: 关注 JIT 热点与版本差异。

### 📊 性能对比
| 方案 | 性能指标 | 适用场景 | 说明 |
|------|---------|---------|------|
| 轻量级锁 | 自旋用户态 | 低/中竞争 | 成本低，避免阻塞 |
| 重量级锁 | 阻塞/唤醒 | 高竞争 | 稳定但切换开销高 |

### ⚠️ 性能陷阱
- **长时间持锁**: 导致自旋浪费或快速膨胀，应缩短临界区。
- **忽视版本差异**: JDK15+ 无偏向，调优策略需更新。

**问题描述**: QPS 较高，计数需要线程安全。

**解决思路**: 
1. 如果仅需简单互斥，使用 synchronized（JDK15 后无偏向，直接轻量）。

**代码实现**:
```java

  private long v;
  private final Object lock = new Object();

}
```

**关键点**: 
- 锁对象固定。
- 竞争高时考虑无锁/分段替代。

**地铁闸机类比**: 偏向锁像私人卡专用闸机；轻量锁像自助闸机自检；自旋失败请安保（重量级）人工放行。JDK15 关闭私人闸机，直接用自助+人工。


## 总结

### ✨ 一句话总结
> **synchronized 通过分层锁应对不同竞争，JDK15 移除偏向锁以减少撤销开销并提升稳定性。**

### 📌 核心记忆点
1. 锁状态：偏向→轻量→重量，只升不降。
2. 偏向撤销成本高，JDK15 默认移除。
3. 低竞争轻量足够，竞争高需缩短临界区或换方案。
