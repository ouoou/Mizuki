## What - 是什么

### 🎯 定义
**上下文切换（Context Switch）**是指 CPU 从一个线程（或进程）切换到另一个线程（或进程）时，需要保存当前线程的状态信息，并加载新线程的状态信息的过程。

简单来说，就是 CPU 从一个任务切换到另一个任务时，需要"记住"当前任务做到哪里了，然后"加载"新任务的状态，以便继续执行。

### 🔍 本质特征
- **状态保存与恢复**：必须保存当前线程的 CPU 寄存器、程序计数器、栈指针等状态
- **时间开销**：上下文切换需要消耗 CPU 时间，属于系统开销
- **透明性**：对应用程序来说是透明的，由操作系统内核调度器完成
- **频繁性**：在多线程环境下，上下文切换会频繁发生

### 📐 基本结构/组成
```
上下文切换包含的主要内容：
┌─────────────────────────────────┐
│  1. CPU 寄存器状态               │
│     - 通用寄存器 (EAX, EBX...)  │
│     - 程序计数器 (PC)            │
│     - 栈指针 (SP)                │
│     - 标志寄存器 (FLAGS)         │
├─────────────────────────────────┤
│  2. 内存管理信息                 │
│     - 页表指针                   │
│     - 内存映射信息               │
├─────────────────────────────────┤
│  3. 线程控制块 (TCB)             │
│     - 线程 ID                    │
│     - 线程状态                   │
│     - 优先级                     │
│     - 栈信息                     │
└─────────────────────────────────┘
```

### 📝 关键术语
| 术语                    | 说明                          |
| --------------------- | --------------------------- |
| **上下文（Context）**      | 线程执行时需要的所有状态信息，包括寄存器、内存映射等  |
| **线程控制块（TCB）**        | 操作系统为每个线程维护的数据结构，存储线程的状态信息  |
| **调度器（Scheduler）**    | 操作系统内核中负责决定哪个线程获得 CPU 时间的组件 |
| **[时间片](../操作系统/什么是时间片？.md)（Time Slice）**   | 分配给线程的 CPU 执行时间片段           |
| **抢占式调度（Preemptive）** | 操作系统可以强制中断当前线程，切换到其他线程      |

## Why - 为什么

### 🧠 设计原理
上下文切换是多任务操作系统的核心机制，其设计原理基于以下考虑：

1. **CPU 资源有限性**：单个 CPU 核心在同一时刻只能执行一个线程
2. **并发需求**：需要让多个线程"看起来"同时执行，提升系统响应性和资源利用率
3. **公平性**：通过时间片轮转，确保所有线程都有机会执行
4. **响应性**：高优先级线程可以抢占低优先级线程，保证重要任务及时响应

### 💡 解决的问题
- **问题1**: **单核 CPU 如何实现多任务并发**
  - 通过快速切换线程，让用户感觉多个任务在同时执行
  
- **问题2**: **如何公平分配 CPU 资源**
  - 通过时间片轮转，确保每个线程都能获得执行机会
  
- **问题3**: **如何响应紧急任务**
  - 通过抢占式调度，高优先级线程可以中断低优先级线程

### ⚡ 优势/优点
1. **实现多任务并发**：让单核 CPU 能够"同时"运行多个线程，提升系统利用率
2. **提高响应性**：用户界面线程可以及时响应，不会因为后台任务阻塞
3. **资源公平分配**：通过调度算法，确保所有线程都有执行机会
4. **支持优先级调度**：重要任务可以优先执行

### ⚠️ 局限性/缺点
- **性能开销**：上下文切换本身需要消耗 CPU 时间（通常几微秒到几十微秒）
- **缓存失效**：切换线程可能导致 CPU 缓存失效，影响性能
- **频繁切换**：过多的上下文切换会降低系统整体性能
- **资源消耗**：需要维护每个线程的状态信息，占用内存

### 🔬 为什么这样设计
上下文切换采用"保存-恢复"模式的原因：

1. **状态完整性**：必须完整保存线程的所有状态，才能保证切换回来后继续正确执行
2. **隔离性**：不同线程的状态必须隔离，避免相互干扰
3. **效率权衡**：虽然切换有开销，但相比让线程等待，这种方式能更好地利用 CPU
4. **硬件支持**：现代 CPU 提供了专门的指令和硬件支持，加速上下文切换过程

## When - 什么时候用

### ✅ 适用场景
1. **多线程并发执行**:
   - 描述：当系统中有多个线程需要共享 CPU 时，必然发生上下文切换
   - 示例：Web 服务器处理多个请求、GUI 应用响应多个用户操作

2. **时间片耗尽**:
   - 描述：当前线程的时间片用完后，操作系统会切换到其他线程
   - 示例：长时间运行的 CPU 密集型任务会被定期中断

3. **线程阻塞**:
   - 描述：线程因为 I/O 操作、等待锁等原因阻塞时，CPU 会切换到其他就绪线程
   - 示例：线程读取文件、等待网络响应时

4. **高优先级线程就绪**:
   - 描述：高优先级线程变为就绪状态时，可以抢占当前线程
   - 示例：用户交互线程可以抢占后台计算线程

5. **线程主动让出 CPU**:
   - 描述：线程调用 `Thread.yield()` 或 `sleep()` 等方法主动让出 CPU
   - 示例：线程在循环中定期让出 CPU，避免占用过多资源

### ❌ 不适用场景
- **单线程执行**：只有一个线程在运行，不需要上下文切换
- **线程已终止**：线程执行完毕，直接销毁，不需要切换回来

### 🎯 判断标准
> **什么时候会发生上下文切换？**
> - 条件1: 系统中有多个线程处于就绪状态
> - 条件2: 当前线程的时间片用完或被阻塞
> - 条件3: 有更高优先级的线程需要执行
> - 条件4: 线程主动让出 CPU

### 📊 与其他方案对比
| 方案           | 适用场景        | 优缺点              | 选择建议          |
| ------------ | ----------- | ---------------- | ------------- |
| **多进程**      | 需要强隔离的场景    | 隔离性好，但切换开销更大     | 需要进程隔离时使用     |
| **协程/纤程**    | 高并发 I/O 场景  | 切换开销小，但需要语言/框架支持 | 高并发场景优先考虑     |
| **上下文切换**    | 通用多线程场景     | 平衡了性能和隔离性        | Java 多线程的标准方式 |
| **单线程+事件循环** | Node.js 等场景 | 无切换开销，但无法利用多核    | I/O 密集型应用     |

## How - 如何使用

### 🔧 基本步骤
上下文切换的过程（由操作系统完成）：

1. **保存当前线程状态**:
   - 将 CPU 寄存器的值保存到线程控制块（TCB）
   - 保存程序计数器、栈指针等关键信息

2. **更新调度信息**:
   - 更新当前线程的状态（运行 → 就绪/阻塞）
   - 从就绪队列中选择下一个要执行的线程

3. **恢复新线程状态**:
   - 从新线程的 TCB 中恢复寄存器值
   - 恢复程序计数器、栈指针等
   - 更新内存管理单元（MMU）的页表

4. **切换到新线程**:
   - 跳转到新线程的程序计数器位置
   - 开始执行新线程

### 💻 代码实现

#### 观察上下文切换的发生
```java
public class ContextSwitchDemo {
    public static void main(String[] args) {
        // 创建多个线程，观察上下文切换
        Thread thread1 = new Thread(() -> {
            for (int i = 0; i < 1000000; i++) {
                // CPU 密集型任务，会频繁发生上下文切换
                Math.sqrt(i);
            }
        });
        
        Thread thread2 = new Thread(() -> {
            for (int i = 0; i < 1000000; i++) {
                Math.sqrt(i);
            }
        });
        
        thread1.start();
        thread2.start();
        
        // 两个线程会竞争 CPU，发生上下文切换
    }
}
```

#### 减少上下文切换的优化
```java
public class ReduceContextSwitch {
    // 1. 使用线程池，避免频繁创建/销毁线程
    private static final ExecutorService executor = 
        Executors.newFixedThreadPool(10);
    
    // 2. 使用无锁数据结构，减少锁竞争
    private final AtomicInteger counter = new AtomicInteger(0);
    
    // 3. 减少线程数量，避免过度并发
    public void processTasks(List<Task> tasks) {
        // 使用合适的线程数，通常为 CPU 核心数
        int threadCount = Runtime.getRuntime().availableProcessors();
        ExecutorService pool = Executors.newFixedThreadPool(threadCount);
        
        tasks.forEach(task -> pool.submit(() -> {
            // 处理任务
            processTask(task);
        }));
    }
    
    // 4. 使用 CAS 操作代替锁
    public void increment() {
        counter.incrementAndGet(); // 无锁操作，减少阻塞
    }
}
```

### 🧩 关键步骤详解
- **保存状态的关键点**: 
  - 必须保存所有 CPU 寄存器的值，包括通用寄存器、程序计数器、栈指针
  - 保存操作必须原子性完成，不能被打断
  
- **选择线程的关键点**:
  - 调度器根据调度算法（如时间片轮转、优先级调度）选择下一个线程
  - 优先选择就绪队列中优先级高的线程

- **恢复状态的关键点**:
  - 恢复寄存器值后，CPU 会从新线程上次中断的地方继续执行
  - 需要更新内存映射，确保新线程访问正确的内存空间

### 📈 复杂度分析
- **时间复杂度**: O(1) - 上下文切换的时间复杂度是常数级的，但实际耗时取决于需要保存/恢复的状态量
- **空间复杂度**: O(1) - 每个线程需要固定的内存空间存储上下文信息
- **分析过程**: 
  - 上下文切换的时间主要消耗在：保存/恢复寄存器（~1-2 微秒）、更新页表（~几微秒）、缓存失效（影响后续执行）
  - 总耗时通常在 1-10 微秒之间，但在高并发场景下，频繁切换会累积成显著开销

## 核心要点

### 🔑 核心思想
> **上下文切换是多任务操作系统的核心机制，通过保存和恢复线程状态，实现 CPU 在多个线程间的快速切换，从而让系统看起来能够并发执行多个任务。**

### 🎯 关键技巧
1. **理解切换时机**：时间片耗尽、线程阻塞、高优先级线程就绪时会发生切换
2. **优化策略**：减少线程数量、使用无锁数据结构、合理使用线程池
3. **性能监控**：使用工具（如 `jstack`、`jstat`）监控上下文切换次数

### ⚠️ 易错点
| 易错点 | 错误示例 | 正确做法 | 原因 |
|--------|----------|----------|------|
| 认为线程越多越好 | 创建 1000 个线程处理任务 | 使用线程池，线程数 = CPU 核心数 | 过多线程导致频繁切换，性能下降 |
| 忽略锁竞争 | 大量线程竞争同一个锁 | 使用无锁数据结构或减少锁粒度 | 锁竞争导致线程阻塞，增加切换 |
| 不理解切换开销 | 认为切换是"免费"的 | 意识到切换有成本，需要优化 | 频繁切换会显著影响性能 |

## 常见错误

### ❌ 错误1: 创建过多线程导致频繁上下文切换
**错误表现**: 创建大量线程（如 1000+），导致 CPU 时间大量浪费在上下文切换上
**错误原因**: 不理解上下文切换的开销，认为线程越多性能越好
**正确做法**: 使用线程池，线程数设置为 CPU 核心数的 1-2 倍
**示例**:
```java
// ❌ 错误写法：创建过多线程
for (int i = 0; i < 10000; i++) {
    new Thread(() -> {
        // 处理任务
    }).start();
}

// ✅ 正确写法：使用线程池
ExecutorService executor = Executors.newFixedThreadPool(
    Runtime.getRuntime().availableProcessors()
);
for (int i = 0; i < 10000; i++) {
    executor.submit(() -> {
        // 处理任务
    });
}
```

### ❌ 错误2: 过度使用锁导致线程阻塞
**错误表现**: 大量线程竞争同一个锁，导致频繁阻塞和唤醒，增加上下文切换
**错误原因**: 没有考虑锁的粒度，或者应该使用无锁数据结构
**正确做法**: 减少锁的粒度，或使用 `ConcurrentHashMap`、`AtomicInteger` 等无锁数据结构
**示例**:
```java
// ❌ 错误写法：粗粒度锁
private final Object lock = new Object();
private int counter = 0;

public void increment() {
    synchronized (lock) {
        counter++; // 简单操作也用锁
    }
}

// ✅ 正确写法：使用原子类
private final AtomicInteger counter = new AtomicInteger(0);

public void increment() {
    counter.incrementAndGet(); // 无锁操作
}
```

### ❌ 错误3: 不理解时间片的概念
**错误表现**: 认为线程会一直执行直到完成
**错误原因**: 不理解抢占式调度的机制
**正确做法**: 理解线程会在时间片用完后被切换，长时间任务应该主动让出 CPU
**示例**:
```java
// ❌ 错误理解：认为线程会一直运行
while (true) {
    // 长时间运行，占用 CPU
    heavyComputation();
}

// ✅ 正确做法：定期让出 CPU
while (true) {
    heavyComputation();
    Thread.yield(); // 主动让出 CPU，减少对其他线程的影响
}
```

## 记忆技巧

### 🧠 记忆口诀
> **"保存状态切线程，时间片尽或阻塞，切换有开销，优化要减少"**

解释：
- **保存状态切线程**：上下文切换需要保存当前线程状态，切换到新线程
- **时间片尽或阻塞**：时间片用完或线程阻塞时会触发切换
- **切换有开销**：上下文切换需要消耗 CPU 时间
- **优化要减少**：应该通过减少线程数、使用无锁结构等方式减少切换

### 📝 思维导图/流程图
```
上下文切换流程：
┌─────────────┐
│ 线程A运行中  │
│ (时间片/阻塞)│
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 保存线程A状态│
│ (寄存器/PC)  │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 选择线程B    │
│ (调度算法)   │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 恢复线程B状态│
│ (寄存器/PC)  │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 线程B开始执行│
└─────────────┘
```

### 🔗 类比理解
**类比：餐厅服务员服务多桌客人**

- **上下文切换** = 服务员从一桌客人切换到另一桌
- **保存状态** = 记住当前桌的订单进度（上到第几道菜了）
- **恢复状态** = 到新桌后，查看这桌的订单，继续服务
- **时间片** = 服务员在每桌服务的时间
- **切换开销** = 服务员在桌间走动的时间（不能服务客人）
- **优化策略** = 减少桌数（减少线程），或让客人自助（无锁设计）

## 实战案例

### 📚 案例1: Web 服务器中的上下文切换优化
**问题描述**: 一个 Web 服务器在处理大量并发请求时，性能下降，CPU 利用率高但吞吐量低
**解决思路**: 
1. 使用 `vmstat` 或 `pidstat` 监控上下文切换次数
2. 发现上下文切换次数异常高（> 10000 次/秒）
3. 分析发现线程池大小设置不合理（1000 个线程）
4. 优化线程池大小，设置为 CPU 核心数的 2 倍

**代码实现**:
```java
// 优化前：线程数过多
ExecutorService executor = Executors.newFixedThreadPool(1000);

// 优化后：根据 CPU 核心数设置
int coreCount = Runtime.getRuntime().availableProcessors();
ExecutorService executor = Executors.newFixedThreadPool(coreCount * 2);
```

**关键点**: 
- 监控上下文切换次数是性能调优的第一步
- 线程数不是越多越好，应该根据 CPU 核心数和任务类型合理设置

### 📚 案例2: 减少锁竞争降低上下文切换
**问题描述**: 多线程计数器性能差，大量线程竞争同一个锁
**解决思路**: 
1. 使用 `AtomicInteger` 代替 `synchronized`
2. 或者使用分段锁，减少锁的粒度

**代码实现**:
```java
// 优化前：使用 synchronized
private int counter = 0;
private final Object lock = new Object();

public void increment() {
    synchronized (lock) {
        counter++;
    }
}

// 优化后：使用原子类
private final AtomicInteger counter = new AtomicInteger(0);

public void increment() {
    counter.incrementAndGet(); // 无锁操作，减少阻塞和切换
}
```

**关键点**: 
- 无锁数据结构可以显著减少线程阻塞，从而减少上下文切换
- 对于简单的原子操作，优先使用 `java.util.concurrent.atomic` 包下的类

### 📚 案例3: 监控和分析上下文切换
**问题描述**: 如何监控 Java 应用的上下文切换情况
**解决思路**: 
1. 使用操作系统工具（Linux: `pidstat -w`）
2. 使用 JVM 工具（`jstack`、`jstat`）
3. 分析[线程状态](线程有几种状态，状态之间的流转是怎样的？.md)，找出阻塞的线程

**代码实现**:
```bash
# Linux 监控上下文切换
pidstat -w -p <pid> 1

# 查看线程状态
jstack <pid> | grep -A 10 "BLOCKED"

# 分析线程转储
jstack <pid> > thread_dump.txt
```

**关键点**: 
- 定期监控上下文切换次数，如果异常高需要优化
- 分析线程转储，找出频繁阻塞的线程和原因

## 总结

### ✨ 一句话总结
> **上下文切换是操作系统在多线程间切换 CPU 执行权时，保存和恢复线程状态的过程，虽然实现了并发，但也会带来性能开销。**

### 📌 核心记忆点
1. **定义**：保存当前线程状态，切换到新线程的过程
2. **时机**：时间片耗尽、线程阻塞、高优先级线程就绪时
3. **开销**：每次切换需要几微秒，频繁切换会显著影响性能
4. **优化**：减少线程数、使用无锁数据结构、合理使用线程池
