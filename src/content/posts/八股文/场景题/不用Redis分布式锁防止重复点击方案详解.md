# 不用Redis分布式锁防止重复点击方案详解

用户重复点击是Web应用中常见的问题。在分布式环境下，通常使用Redis分布式锁来解决，但如果系统没有Redis或者不想依赖Redis，应该如何防止用户重复点击？这个问题看似简单，但在高并发、分布式环境下，需要考虑很多技术细节和潜在问题。本文将从业务场景分析出发，深入解析多种不使用Redis分布式锁的实现方案，详细讲解每个方案的优缺点和可能遇到的坑，并提供清晰的流程图。

## 一、业务场景分析

### 1.1 核心需求

**基本需求：**
```
1. 防止用户重复点击导致重复提交
2. 不能使用Redis分布式锁
3. 需要支持分布式环境
4. 需要保证数据一致性
5. 需要保证系统性能
```

### 1.2 重复点击场景

**常见重复点击场景：**
```
1. 提交订单
   - 用户点击"提交订单"按钮
   - 网络延迟，用户以为没反应
   - 再次点击，导致重复下单

2. 支付操作
   - 用户点击"支付"按钮
   - 等待支付结果时重复点击
   - 可能导致重复扣款

3. 表单提交
   - 用户填写表单后提交
   - 提交后重复点击
   - 导致重复数据

4. 点赞/收藏
   - 用户快速连续点击
   - 导致重复操作
   - 数据不一致
```

### 1.3 重复点击的危害

**主要危害：**
```
1. 数据重复
   - 重复订单
   - 重复扣款
   - 重复数据记录

2. 业务异常
   - 库存扣减错误
   - 积分重复发放
   - 优惠券重复使用

3. 用户体验
   - 用户困惑
   - 需要客服处理
   - 影响用户信任

4. 系统压力
   - 重复请求增加系统负载
   - 数据库压力增大
   - 可能引发其他问题
```

### 1.4 技术挑战

**主要挑战：**
```
1. 分布式环境
   - 多台服务器
   - 无法使用本地锁
   - 需要分布式协调

2. 高并发场景
   - 大量并发请求
   - 性能要求高
   - 不能成为瓶颈

3. 数据一致性
   - 保证数据不重复
   - 保证业务逻辑正确
   - 不能丢失数据

4. 用户体验
   - 不能影响正常操作
   - 响应时间要快
   - 错误提示要清晰
```

## 二、方案一：数据库唯一索引

### 2.1 方案概述

**实现思路：**
```
1. 在数据库表中添加唯一索引
2. 唯一索引字段可以是：
   - 业务唯一标识（如订单号）
   - 用户ID + 业务标识组合
   - 请求ID（UUID）
3. 插入数据时，如果违反唯一约束，数据库会抛出异常
4. 捕获异常，判断为重复提交
```

### 2.2 流程图

```
数据库唯一索引防重复流程：
┌─────────────┐
│ 用户点击提交 │
└──────┬──────┘
       │
       ▼
┌─────────────────────┐
│ 生成唯一标识         │
│ - 订单号            │
│ - 用户ID+时间戳     │
│ - UUID              │
└──────┬──────────────┘
       │
       ▼
┌─────────────────────┐
│ 插入数据库           │
│ INSERT INTO ...     │
└──┬──────────────┬───┘
   │成功          │失败（唯一约束）
   ▼              ▼
┌─────────┐  ┌─────────────┐
│继续处理  │  │捕获异常      │
│业务逻辑  │  │判断为重复提交│
└─────────┘  └──────┬───────┘
                    │
                    ▼
            ┌───────────────┐
            │ 返回重复提交提示│
            └───────────────┘
```

### 2.3 优点

**优势：**
```
1. 实现简单
   - 只需要添加唯一索引
   - 不需要额外的中间件
   - 代码改动小

2. 可靠性高
   - 数据库保证唯一性
   - 不会出现重复数据
   - 原子性保证

3. 性能好
   - 数据库索引查询快
   - 不需要额外的网络请求
   - 对应用层透明

4. 通用性强
   - 适用于各种数据库
   - 不依赖外部服务
   - 易于理解和维护
```

### 2.4 缺点

**劣势：**
```
1. 需要设计唯一字段
   - 需要选择合适的唯一字段
   - 可能需要组合多个字段
   - 字段设计不合理会影响性能

2. 错误处理
   - 需要捕获数据库异常
   - 需要区分唯一约束异常和其他异常
   - 错误信息可能不够友好

3. 扩展性
   - 如果业务变化，可能需要修改索引
   - 索引过多会影响写入性能
   - 不适合频繁变化的业务
```

### 2.5 可能遇到的坑

**坑点1：唯一字段选择不当**
```
问题：
- 选择的唯一字段不够唯一
- 不同用户可能生成相同的标识
- 导致误判为重复提交

解决方案：
1. 使用全局唯一标识
   - UUID
   - 雪花算法生成的ID
   - 业务唯一标识（如订单号）

2. 组合多个字段
   - 用户ID + 业务标识 + 时间戳
   - 确保全局唯一

3. 验证唯一性
   - 在应用层验证
   - 确保生成的标识唯一
```

**坑点2：异常处理不当**
```
问题：
- 捕获所有数据库异常
- 无法区分唯一约束异常和其他异常
- 可能误判为重复提交

解决方案：
1. 精确捕获异常
   - 只捕获唯一约束异常
   - 其他异常正常抛出
   - 使用异常类型判断

2. 错误码区分
   - 数据库返回明确的错误码
   - 根据错误码判断
   - 提供友好的错误信息

3. 日志记录
   - 记录异常详情
   - 便于排查问题
   - 区分真正的重复提交和系统异常
```

**坑点3：索引性能问题**
```
问题：
- 唯一索引过多
- 影响写入性能
- 索引维护成本高

解决方案：
1. 合理设计索引
   - 只对必要的字段建唯一索引
   - 避免冗余索引
   - 考虑索引顺序

2. 使用组合索引
   - 多个字段组合成唯一索引
   - 减少索引数量
   - 提高查询效率

3. 定期优化
   - 监控索引使用情况
   - 删除不用的索引
   - 优化索引结构
```

## 三、方案二：数据库乐观锁

### 3.1 方案概述

**实现思路：**
```
1. 在数据库表中添加版本号字段（version）
2. 读取数据时，获取当前版本号
3. 更新数据时，检查版本号是否变化
4. 如果版本号变化，说明数据已被修改，更新失败
5. 更新成功时，版本号自增
```

### 3.2 流程图

```
数据库乐观锁防重复流程：
┌─────────────┐
│ 用户点击提交 │
└──────┬──────┘
       │
       ▼
┌─────────────────────┐
│ 查询数据             │
│ SELECT * FROM ...   │
│ WHERE id = ?        │
│ 获取version字段      │
└──────┬──────────────┘
       │
       ▼
┌─────────────────────┐
│ 执行业务逻辑         │
│ 准备更新数据         │
└──────┬──────────────┘
       │
       ▼
┌─────────────────────┐
│ 更新数据             │
│ UPDATE ... SET ...  │
│ WHERE id = ?        │
│ AND version = ?     │
└──┬──────────────┬───┘
   │更新成功      │更新失败
   │(影响行数>0)  │(影响行数=0)
   ▼              ▼
┌─────────┐  ┌─────────────┐
│继续处理  │  │判断为重复提交│
│version++│  │返回重复提示  │
└─────────┘  └─────────────┘
```

### 3.3 优点

**优势：**
```
1. 性能好
   - 不需要加锁
   - 并发性能高
   - 不会阻塞其他请求

2. 实现简单
   - 只需要添加版本号字段
   - 更新时检查版本号
   - 代码逻辑清晰

3. 适合读多写少
   - 读操作不受影响
   - 写冲突较少时效率高
   - 减少锁竞争
```

### 3.4 缺点

**劣势：**
```
1. 冲突时需要重试
   - 更新失败需要重新读取
   - 可能需要多次重试
   - 增加系统复杂度

2. 不适合写冲突频繁的场景
   - 如果写冲突频繁，重试次数多
   - 性能会下降
   - 用户体验差

3. 需要业务层处理
   - 需要判断更新结果
   - 需要实现重试逻辑
   - 增加代码复杂度
```

### 3.5 可能遇到的坑

**坑点1：版本号字段设计不当**
```
问题：
- 版本号字段类型选择不当
- 版本号溢出
- 版本号更新逻辑错误

解决方案：
1. 使用合适的数据类型
   - 使用INT或BIGINT
   - 考虑溢出问题
   - 初始值设为0或1

2. 版本号更新逻辑
   - 更新时version = version + 1
   - 确保原子性
   - 使用数据库函数

3. 版本号重置
   - 考虑是否需要重置
   - 重置时的业务逻辑
   - 避免版本号冲突
```

**坑点2：重试逻辑不当**
```
问题：
- 重试次数过多
- 重试间隔不合理
- 无限重试导致死循环

解决方案：
1. 设置重试上限
   - 最多重试3-5次
   - 超过上限返回错误
   - 避免无限重试

2. 重试间隔
   - 使用指数退避
   - 避免频繁重试
   - 减少系统压力

3. 重试条件
   - 只对版本冲突重试
   - 其他错误不重试
   - 记录重试日志
```

**坑点3：ABA问题**
```
问题：
- 版本号可能回到原值
- 虽然版本号相同，但数据已变化
- 可能导致数据不一致

解决方案：
1. 使用时间戳
   - 结合时间戳判断
   - 时间戳只能递增
   - 避免ABA问题

2. 使用业务状态
   - 结合业务状态判断
   - 状态变化不可逆
   - 更可靠的判断

3. 使用其他唯一标识
   - 结合其他字段判断
   - 确保数据一致性
   - 多重校验
```

## 四、方案三：数据库悲观锁

### 4.1 方案概述

**实现思路：**
```
1. 使用数据库的SELECT FOR UPDATE
2. 在事务中先查询并加锁
3. 检查数据状态，判断是否已处理
4. 如果未处理，执行业务逻辑并更新状态
5. 如果已处理，返回重复提交提示
6. 提交事务，释放锁
```

### 4.2 流程图

```
数据库悲观锁防重复流程：
┌─────────────┐
│ 用户点击提交 │
└──────┬──────┘
       │
       ▼
┌─────────────────────┐
│ 开启事务             │
│ BEGIN TRANSACTION   │
└──────┬──────────────┘
       │
       ▼
┌─────────────────────┐
│ 查询并加锁           │
│ SELECT * FROM ...   │
│ WHERE id = ?        │
│ FOR UPDATE          │
└──────┬──────────────┘
       │
       ▼
┌─────────────────────┐
│ 检查数据状态         │
│ - 是否已处理？       │
│ - 状态是什么？       │
└──┬──────────────┬───┘
   │未处理        │已处理
   ▼              ▼
┌─────────┐  ┌─────────────┐
│执行业务  │  │回滚事务      │
│更新状态  │  │返回重复提示  │
│提交事务  │  └─────────────┘
└─────────┘
```

### 4.3 优点

**优势：**
```
1. 可靠性高
   - 数据库保证一致性
   - 不会出现并发问题
   - 数据安全

2. 实现简单
   - 使用数据库原生锁
   - 代码逻辑清晰
   - 易于理解

3. 适合写冲突频繁的场景
   - 直接加锁，避免重试
   - 保证数据一致性
   - 减少冲突
```

### 4.4 缺点

**劣势：**
```
1. 性能较差
   - 需要加锁，阻塞其他请求
   - 并发性能低
   - 可能成为瓶颈

2. 死锁风险
   - 多个事务加锁顺序不一致
   - 可能导致死锁
   - 需要处理死锁

3. 事务时间长
   - 锁持有时间长
   - 影响其他请求
   - 需要优化事务范围
```

### 4.5 可能遇到的坑

**坑点1：死锁问题**
```
问题：
- 多个事务加锁顺序不一致
- 导致死锁
- 数据库自动回滚，影响业务

解决方案：
1. 统一加锁顺序
   - 按照固定顺序加锁
   - 避免循环等待
   - 减少死锁概率

2. 设置锁超时
   - 设置锁等待超时时间
   - 超时后自动释放
   - 避免长时间等待

3. 死锁检测
   - 数据库自动检测死锁
   - 自动回滚一个事务
   - 应用层重试
```

**坑点2：锁范围过大**
```
问题：
- 锁的范围太大
- 影响其他正常请求
- 系统性能下降

解决方案：
1. 缩小锁范围
   - 只锁必要的行
   - 使用精确的WHERE条件
   - 避免全表锁

2. 缩短事务时间
   - 尽快释放锁
   - 减少事务中的业务逻辑
   - 异步处理非关键操作

3. 使用行锁
   - 使用SELECT FOR UPDATE
   - 只锁查询到的行
   - 避免表锁
```

**坑点3：锁超时处理**
```
问题：
- 锁等待超时
- 事务回滚
- 用户请求失败

解决方案：
1. 设置合理的超时时间
   - 根据业务特点设置
   - 不能太短也不能太长
   - 平衡用户体验和系统性能

2. 超时后的处理
   - 返回友好的错误信息
   - 提示用户稍后重试
   - 记录超时日志

3. 重试机制
   - 前端自动重试
   - 后端重试逻辑
   - 避免用户重复点击
```

## 五、方案四：Token机制

### 5.1 方案概述

**实现思路：**
```
1. 用户访问页面时，生成一个唯一的Token
2. Token存储在Session或返回给前端
3. 用户提交时，携带Token
4. 服务端验证Token是否存在
5. 如果Token存在，处理请求并删除Token
6. 如果Token不存在，判断为重复提交
```

### 5.2 流程图

```
Token机制防重复流程：
┌─────────────┐
│ 用户访问页面 │
└──────┬──────┘
       │
       ▼
┌─────────────────────┐
│ 生成唯一Token        │
│ - UUID              │
│ - 时间戳+随机数      │
└──────┬──────────────┘
       │
       ▼
┌─────────────────────┐
│ 存储Token            │
│ - Session中         │
│ - 返回给前端         │
└──────┬──────────────┘
       │
       ▼
┌─────────────┐
│ 用户点击提交 │
│ 携带Token    │
└──────┬──────┘
       │
       ▼
┌─────────────────────┐
│ 验证Token            │
│ - Token是否存在？    │
│ - Token是否有效？    │
└──┬──────────────┬───┘
   │有效          │无效
   ▼              ▼
┌─────────┐  ┌─────────────┐
│删除Token │  │返回重复提示  │
│处理请求  │  └─────────────┘
└─────────┘
```

### 5.3 优点

**优势：**
```
1. 实现简单
   - 前端后端配合即可
   - 不需要额外的中间件
   - 代码改动小

2. 用户体验好
   - 前端可以控制
   - 可以防止误操作
   - 错误提示清晰

3. 性能好
   - 验证逻辑简单
   - 不需要数据库查询
   - 响应速度快
```

### 5.4 缺点

**劣势：**
```
1. 需要Session支持
   - 依赖Session存储
   - 分布式环境下需要Session共享
   - 增加系统复杂度

2. 前端可能绕过
   - 恶意用户可能绕过
   - 需要后端验证
   - 安全性依赖实现

3. Token管理
   - Token需要及时清理
   - 过期Token需要处理
   - 增加维护成本
```

### 5.5 可能遇到的坑

**坑点1：Session共享问题**
```
问题：
- 分布式环境下Session不共享
- Token存储在不同服务器
- 导致验证失败

解决方案：
1. Session共享
   - 使用Redis存储Session
   - 使用数据库存储Session
   - 使用Session粘性

2. 数据库存储Token
   - Token存储在数据库
   - 所有服务器共享
   - 需要定期清理

3. 无状态Token
   - 使用JWT等无状态Token
   - 包含签名验证
   - 不需要存储
```

**坑点2：Token过期处理**
```
问题：
- Token过期后无法使用
- 用户需要刷新页面
   - 用户体验差

解决方案：
1. 设置合理的过期时间
   - 根据业务特点设置
   - 不能太短也不能太长
   - 平衡安全性和用户体验

2. Token刷新机制
   - 过期前自动刷新
   - 前端定时刷新
   - 后端返回新Token

3. 过期后的处理
   - 返回友好的错误信息
   - 提示用户刷新页面
   - 自动跳转或刷新
```

**坑点3：Token泄露风险**
```
问题：
- Token可能被泄露
   - 被恶意使用
   - 安全性问题

解决方案：
1. Token加密
   - 使用加密算法
   - 包含签名验证
   - 防止篡改

2. 绑定用户信息
   - Token包含用户ID
   - 验证时检查用户
   - 防止跨用户使用

3. 一次性Token
   - Token使用后立即删除
   - 不能重复使用
   - 提高安全性
```

## 六、方案五：幂等性设计

### 6.1 方案概述

**实现思路：**
```
1. 设计接口为幂等的
2. 相同请求多次执行，结果相同
3. 使用唯一标识判断是否已处理
4. 已处理直接返回结果
5. 未处理执行业务逻辑
```

### 6.2 流程图

```
幂等性设计防重复流程：
┌─────────────┐
│ 用户点击提交 │
│ 携带唯一标识 │
└──────┬──────┘
       │
       ▼
┌─────────────────────┐
│ 查询处理记录         │
│ SELECT * FROM ...   │
│ WHERE idempotent_id │
│ = ?                 │
└──┬──────────────┬───┘
   │已处理        │未处理
   ▼              ▼
┌─────────┐  ┌─────────────┐
│返回结果  │  │执行业务逻辑  │
│（幂等）  │  │保存处理记录  │
└─────────┘  │返回结果      │
             └─────────────┘
```

### 6.3 优点

**优势：**
```
1. 设计优雅
   - 符合RESTful设计原则
   - 接口天然支持重复调用
   - 易于理解和维护

2. 可靠性高
   - 多次调用结果一致
   - 不会产生副作用
   - 数据一致性保证

3. 适合分布式
   - 不依赖外部服务
   - 可以跨服务调用
   - 易于扩展
```

### 6.4 缺点

**劣势：**
```
1. 需要设计唯一标识
   - 需要选择合适的标识
   - 可能需要组合多个字段
   - 增加设计复杂度

2. 需要存储处理记录
   - 需要额外的存储空间
   - 需要定期清理
   - 增加维护成本

3. 业务逻辑复杂
   - 需要判断是否已处理
   - 需要处理各种边界情况
   - 代码复杂度增加
```

### 6.5 可能遇到的坑

**坑点1：唯一标识设计不当**
```
问题：
- 唯一标识不够唯一
- 不同请求生成相同标识
- 导致误判

解决方案：
1. 使用全局唯一标识
   - UUID
   - 雪花算法ID
   - 业务唯一标识

2. 组合多个字段
   - 用户ID + 业务标识 + 时间戳
   - 确保全局唯一
   - 提高可靠性

3. 验证唯一性
   - 应用层验证
   - 数据库唯一索引
   - 双重保障
```

**坑点2：处理记录存储问题**
```
问题：
- 处理记录过多
- 存储空间不足
- 查询性能下降

解决方案：
1. 定期清理
   - 清理过期的处理记录
   - 只保留必要的记录
   - 使用定时任务

2. 分表存储
   - 按时间分表
   - 按业务分表
   - 提高查询性能

3. 使用缓存
   - 热点数据缓存
   - 减少数据库查询
   - 提高性能
```

**坑点3：幂等性边界不清**
```
问题：
- 哪些操作需要幂等
- 幂等的粒度是什么
- 边界情况处理

解决方案：
1. 明确幂等范围
   - 定义哪些接口需要幂等
   - 明确幂等的粒度
   - 文档化

2. 统一处理逻辑
   - 使用统一的幂等框架
   - 减少重复代码
   - 提高可维护性

3. 测试验证
   - 编写幂等性测试
   - 验证各种场景
   - 确保正确性
```

## 七、方案六：消息队列去重

### 7.1 方案概述

**实现思路：**
```
1. 请求先发送到消息队列
2. 消息队列根据唯一标识去重
3. 相同的消息只处理一次
4. 消费者处理消息
5. 保证消息处理的幂等性
```

### 7.2 流程图

```
消息队列去重防重复流程：
┌─────────────┐
│ 用户点击提交 │

       │
       ▼

│ - 业务唯一标识       │
│ - UUID              │
└──────┬──────────────┘
       │
       ▼
┌─────────────────────┐

└──────┬──────────────┘
       │
       ▼

       ▼
```

### 7.4 缺点

```
   - 不能保证强一致性
```

### 7.5 可能遇到的坑

```
解决方案：

   - 支持消息去重
   - 保证消息唯一性
   - 提供去重配置

2. 业务层二次去重
   - 消息队列去重 + 业务层去重
   - 双重保障
   - 提高可靠性

3. 监控和告警
   - 监控消息处理情况
   - 发现异常及时告警
   - 快速处理问题
```

**坑点2：消息处理失败**

```
问题：
```

## 八、方案七：前端防抖节流

### 8.1 方案概述

**实现思路：**
```
1. 前端使用防抖（Debounce）
   - 连续点击只执行最后一次
   - 延迟执行，避免频繁触发

2. 前端使用节流（Throttle）
   - 限制执行频率
   - 固定时间间隔执行一次

3. 按钮禁用
   - 点击后立即禁用按钮
   - 请求完成后启用
   - 防止重复点击
```

### 8.2 流程图

```
前端防抖节流防重复流程：
┌─────────────┐
│ 用户点击按钮 │
└──────┬──────┘
       │
       ▼
┌─────────────────────┐
│ 防抖处理             │
│ - 清除之前的定时器   │
│ - 设置新的定时器     │
│ - 延迟执行           │
└──┬──────────────┬───┘
   │时间到        │再次点击
   ▼              ▼
┌─────────┐  ┌─────────────┐
│执行请求  │  │重置定时器   │
│禁用按钮  │  └─────────────┘
└────┬────┘
     │
     ▼
┌─────────────────────┐
│ 请求完成             │
│ - 启用按钮           │
│ - 显示结果           │
└─────────────────────┘
```

### 8.3 优点

**优势：**
```
1. 用户体验好
   - 前端立即响应
   - 按钮状态清晰
   - 减少无效请求

2. 实现简单
   - 前端代码即可实现
   - 不需要后端配合
   - 代码改动小

3. 减少服务器压力
   - 减少无效请求
   - 降低服务器负载
   - 提高系统性能
```

### 8.4 缺点

**劣势：**
```
1. 可以被绕过
   - 恶意用户可能绕过
   - 需要后端验证
   - 不能完全依赖

2. 网络问题
   - 网络延迟时可能失效
   - 需要配合后端
   - 不能单独使用

3. 兼容性
   - 不同浏览器可能不同
   - 需要测试
   - 增加维护成本
```

### 8.5 可能遇到的坑

**坑点1：防抖时间设置不当**
```
问题：
- 防抖时间太短，仍然可能重复
- 防抖时间太长，用户体验差
- 不同场景需要不同时间

解决方案：
1. 根据业务设置
   - 提交操作：500ms-1s
   - 搜索操作：300ms
   - 按钮点击：200ms

2. 动态调整
   - 根据网络情况调整
   - 根据业务特点调整
   - 提供配置选项

3. 用户反馈
   - 显示加载状态
   - 提示用户等待
   - 改善用户体验
```

**坑点2：按钮状态管理**
```
问题：
- 按钮状态不同步
- 请求失败后按钮未恢复
- 用户体验差

解决方案：
1. 统一状态管理
   - 使用状态机
   - 统一管理按钮状态
   - 确保状态一致

2. 异常处理
   - 请求失败恢复按钮
   - 网络错误恢复按钮
   - 超时恢复按钮

3. 用户提示
   - 显示加载状态
   - 显示错误信息
   - 提供重试选项
```

**坑点3：多标签页问题**
```
问题：
- 多个标签页同时操作
- 前端防抖无法跨标签页
- 仍然可能重复提交

解决方案：
1. 后端验证
   - 前端防抖 + 后端验证
   - 双重保障
   - 提高可靠性

2. 使用SharedWorker
   - 跨标签页共享状态
   - 统一防抖控制
   - 提高一致性

3. 使用LocalStorage
   - 跨标签页共享状态
   - 简单实现
   - 需要注意清理
```

### 11.4 常见问题FAQ

**Q1: 为什么不能只用前端防抖？**
```
A: 
1. 前端可以被绕过
   - 恶意用户可能绕过
   - 需要后端验证
   - 不能完全依赖

2. 网络问题
   - 网络延迟时可能失效
   - 需要配合后端
   - 双重保障更可靠
```

**Q2: 数据库唯一索引会影响性能吗？**
```
A: 
1. 影响很小
   - 索引查询很快
   - 写入时检查索引
   - 对性能影响很小

2. 可以优化
   - 合理设计索引
   - 避免冗余索引
   - 定期优化
```

**Q3: 乐观锁和悲观锁如何选择？**
```
A: 
1. 读多写少：乐观锁
   - 性能好
   - 适合读多写少

2. 写冲突频繁：悲观锁
   - 可靠性高
   - 适合写冲突多

3. 根据实际情况选择
   - 分析业务特点
   - 测试验证
   - 选择合适方案
```

**Q4: 如何保证分布式环境下的可靠性？**
```
A: 
1. 使用数据库方案
   - 数据库唯一索引
   - 数据库锁
   - 所有服务器共享

2. 使用共享存储
   - Session共享
   - 数据库存储
   - 消息队列

3. 无状态设计
   - 使用JWT等
   - 不依赖本地状态
   - 易于扩展
```

## 十二、总结

不使用Redis分布式锁防止用户重复点击，有多种可行的方案。本文从业务场景分析出发，详细讲解了：

1. **数据库唯一索引**：实现简单，可靠性高，适合大多数场景
2. **数据库乐观锁**：性能好，适合读多写少
3. **数据库悲观锁**：可靠性高，适合写冲突频繁
4. **Token机制**：用户体验好，适合Web应用
5. **幂等性设计**：设计优雅，适合分布式系统
6. **消息队列去重**：解耦削峰，适合异步处理
7. **前端防抖节流**：用户体验好，需要配合后端

**关键要点：**
- 多层防护：前端 + 后端双重保障，提高可靠性
- 根据场景选择：分析业务特点，选择合适的方案
- 性能优先：不影响正常请求，优化防重复逻辑
- 用户体验：快速响应，友好提示，提供重试

**推荐方案：**
- 简单场景：前端防抖 + 数据库唯一索引
- 复杂场景：前端防抖 + Token机制 + 幂等性设计
- 高并发：前端防抖 + 乐观锁 + 幂等性设计

**最终目标：**
在不使用Redis分布式锁的情况下，通过合适的方案组合，实现：
- 防止重复提交（可靠性99.9%）
- 性能影响最小（响应时间增加 < 10%）
- 用户体验良好（快速响应，友好提示）
- 易于维护（代码清晰，易于理解）
