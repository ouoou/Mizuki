# 秒杀系统设计详解

设计一个秒杀系统，是典型的高并发场景挑战。在短时间内（如1秒内）有数万甚至数十万用户同时抢购少量商品（如100件），系统需要承受巨大的流量冲击：高并发、高可用、数据一致性、防刷、防超卖等。如何在有限的资源下设计一个稳定可靠的秒杀系统？本文将从业务场景分析出发，深入解析秒杀系统设计的完整思路，详细讲解每个设计决策和可能遇到的坑，并提供清晰的流程图。

## 一、业务场景分析

### 1.1 核心需求

**基本需求：**
```
1. 商品数量有限（如100件）
2. 大量用户同时抢购（如10万用户）
3. 秒杀时间短（如1秒内开始）
4. 需要保证公平性（先到先得）
5. 需要防止超卖（不能卖出超过库存）
6. 需要防止刷单（一人只能买一件）
7. 需要高可用（不能宕机）
8. 需要快速响应（用户体验好）
```

### 1.2 业务特点分析

**秒杀业务特点：**
```
1. 流量特点
   - 流量瞬间爆发（如0点开始）
   - QPS可能达到10万+
   - 持续时间短（几分钟）
   - 流量分布不均

2. 商品特点
   - 商品数量少（如100件）
   - 价格优惠大
   - 用户需求强烈
   - 竞争激烈

3. 用户特点
   - 用户数量多（如10万+）
   - 用户行为相似（同时点击）
   - 用户期望高（必须抢到）
   - 用户可能使用脚本

4. 系统特点
   - 读多写少（查看商品 > 下单）
   - 写操作集中在秒杀开始
   - 需要强一致性（库存不能超卖）
   - 需要高性能（快速响应）
```

### 1.3 技术挑战

**主要挑战：**
```
1. 高并发
   - 需要支持10万+ QPS
   - 系统不能崩溃
   - 需要快速响应

2. 高可用
   - 服务不能宕机
   - 需要容错机制
   - 需要快速恢复

3. 数据一致性
   - 库存不能超卖
   - 一人只能买一件
   - 订单数据准确

4. 防刷防作弊
   - 防止脚本刷单
   - 防止重复购买
   - 保证公平性

5. 性能优化
   - 减少数据库压力
   - 减少网络延迟
   - 提高响应速度
```

### 1.4 性能指标

**关键指标：**
```
1. QPS要求
   - 峰值QPS：10万+
   - 平均QPS：1万-5万
   - 持续时间：几分钟

2. 响应时间
   - 页面加载：< 100ms
   - 下单响应：< 500ms
   - 支付响应：< 1s

3. 可用性
   - 系统可用性：99.9%
   - 秒杀成功率：> 95%
   - 数据准确性：100%

4. 资源限制
   - 服务器资源有限
   - 数据库连接有限
   - 网络带宽有限
```

```
1. 减少无效请求

3. 提升用户体验
```

**静态化策略：**

```
1. 秒杀页面静态化

2. 商品信息缓存

   - 减少后端请求

3. 动态数据分离
   - 静态部分：CDN
   - 动态部分：API接口
   - 减少服务器压力

```

```
   - 限制单用户QPS（如5）
   - 防止脚本刷单
   - 提高安全性

4. 接口限流
   - 不同接口不同限流
   - 秒杀接口严格限流
   - 其他接口宽松限流
```

### 5.3 库存预扣设计

```
2. 预扣操作
   - 使用DECR原子操作
   - 检查返回值是否 >= 0
   - 负数表示库存不足

3. 回滚机制
   - 订单失败时回滚库存
   - 使用INCR恢复库存
   - 保证数据一致性
```

### 5.4 防重复购买

**防重复购买策略：**
```
1. Redis标记
   - Key: seckill:user:{userId}:{productId}
   - Value: 订单号或时间戳
   - 过期时间: 秒杀活动结束时间

2. 数据库唯一索引
   - 用户ID + 商品ID唯一索引
   - 数据库层面保证
   - 双重保障

3. 分布式锁
   - 用户ID + 商品ID作为锁
   - 保证同一用户同一商品只买一次
   - 防止并发问题
```

## 九、可能遇到的坑

### 9.1 坑点1：库存超卖问题

**问题描述：**
```
现象：
- 库存被超卖
- 卖出数量超过实际库存
- 用户下单成功但无货可发

原因：
- 并发情况下库存校验不准确
- 数据库更新不是原子操作
- Redis和数据库数据不一致
```

**解决方案：**
```
1. Redis原子操作
   - 使用DECR原子操作
   - 检查返回值
   - 负数表示库存不足

2. 数据库唯一约束
   - 用户ID + 商品ID唯一索引
   - 数据库层面保证
   - 防止重复购买

3. 双重校验
   - Redis预扣 + 数据库校验
   - 双重保障
   - 提高可靠性

4. 库存回滚
   - 订单失败时回滚库存
   - 保证数据一致性
   - 避免库存丢失
```

**预防措施：**
```
- 使用Redis原子操作
- 数据库唯一约束
- 双重校验
- 库存回滚机制
- 监控库存变化
```

```
3. 异步处理
```

### 9.3 坑点3：缓存穿透和击穿

**问题描述：**
```

1. 缓存穿透
   - 缓存空值
   - 布隆过滤器
   - 参数校验

2. 缓存击穿
   - 热点数据永不过期
   - 互斥锁保护
   - 异步更新

3. 缓存雪崩
   - 过期时间随机化
   - 多级缓存
   - 限流保护
```

**预防措施：**
```
- 缓存空值

- 多级缓存
```
