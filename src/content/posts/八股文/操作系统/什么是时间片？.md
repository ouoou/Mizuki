## What - 是什么

### 🎯 定义
**时间片（Time Slice/Quantum）** 是操作系统在抢占式调度中分配给可运行进程或线程的一小段 CPU 执行时间。时间片用完后，调度器会触发上下文切换，将 CPU 让给下一个就绪任务。

### 🔍 本质特征
- **抢占式**：到达时间片时钟中断点会强制切换，保证公平性
- **颗粒度可调**：时间片长短影响响应性与吞吐量的权衡
- **配合调度算法**：常与时间片轮转、优先级调度联合使用
- **与硬件时钟耦合**：依赖定时器中断驱动

### 📐 基本结构/组成
```
应用线程 → 就绪队列 → 获得时间片 → 计时器递减
                               │
                               ├─ 时间片耗尽 → 上下文切换
                               └─ 主动让出/阻塞 → 重新调度
```

### 📝 关键术语
| 术语     | 说明                    |
| ------ | --------------------- |
| 时间片/量子 | 可运行任务本轮获得的 CPU 时间预算   |
| 时间片轮转  | 每个任务按固定时间片循环执行的调度策略   |
| 抢占式调度  | 内核可在时间片到期时中断当前任务      |
| 时钟中断   | 由硬件定时器触发，用于递减时间片与驱动调度 |
| 上下文切换  | 时间片耗尽或阻塞时保存/恢复任务状态    |

## Why - 为什么

### 🧠 设计原理
- 让单核也能通过快速切换实现并发感知
- 用固定量化的时间份额实现 CPU 资源公平分配
- 结合优先级与阻塞事件，兼顾响应性和吞吐量

### 💡 解决的问题
- **公平性**：避免单个任务长期占用 CPU
- **响应性**：交互式任务能在短时间内获得执行
- **可预期性**：调度器可基于时间片做配额和统计

### ⚡ 优势/优点
1. **保证公平**：所有就绪任务都有执行机会
2. **提升交互体验**：时间片较短时，前端/IO 线程响应快
3. **实现简单**：与时钟中断结合即可落地

### ⚠️ 局限性/缺点
- 时间片太短：上下文切换开销上升，吞吐下降
- 时间片太长：交互卡顿，响应延迟增加
- 不能直接满足硬实时场景的严格时限

### 🔬 为什么这样设计
时间片把 CPU 时间量化并引入抢占点，使调度器能在简单规则下平衡公平性与响应性，同时保留对不同负载通过"调时间片 + 调算法"的调优空间。

## When - 什么时候用

### ✅ 适用场景
1. **通用多任务操作系统**：桌面、移动、服务器的普通进程调度
2. **交互型/IO 密集工作负载**：需要较低输入到反馈延迟
3. **多租户/容器场景**：避免单租户长时间独占 CPU

### ❌ 不适用场景
- **硬实时控制**：需要确定性延迟，通常改用固定优先级+抢占或 EDF
- **长批处理单任务**：可增大时间片或使用批处理队列减少切换开销

### 🎯 判断标准
> 选择时间片配置时，关注：
> - 交互延迟目标（常见 1~10ms）
> - 切换开销占比（切换成本 / 时间片时长）
> - 负载类型（IO 密集 vs CPU 密集）

## How - 如何使用

### 🔧 基本步骤
1. **设置时间片长度**：根据硬件时钟频率与负载设定，如 4ms/8ms
2. **调度入口**：时钟中断递减时间片；耗尽或主动让出时进入调度器
3. **选择下一个任务**：依据算法（时间片轮转/优先级）挑选就绪任务
4. **执行与计时**：任务运行并消耗剩余时间片，直到耗尽或阻塞

### 💻 代码实现（伪代码）
```c
// 伪代码：抢占式时间片调度循环
on_timer_tick() {
  current.timeslice--;
  if (current.timeslice == 0 || current.state != RUNNING) {
    save_context(current);
    enqueue_ready(current);
    next = pick_next_ready();
    next.timeslice = DEFAULT_SLICE;
    switch_to(next);
  }
}
```

### 🧩 关键步骤详解
- **时间片递减**：由时钟中断驱动，保证抢占点可靠出现
- **耗尽处理**：必须先保存上下文，再入队，避免状态丢失
- **重置时间片**：新选中任务上 CPU 前重新初始化时间片

### 📈 复杂度分析
- **调度决策**：取决于就绪队列数据结构（O(1) 的多级队列常见）
- **切换开销**：≈ 保存/恢复寄存器 + 缓存失效，需控制切换频率

## 核心要点

### 🔑 核心思想
> 用可调的量化时间份额驱动抢占式调度，在公平与响应之间找到平衡。

### 🎯 关键技巧
1. **按负载调时间片**：交互小、批处理大，配合优先级衰减更稳
2. **避免抖动**：保持时钟中断稳定，避免过度动态调节

### ⚠️ 易错点
| 易错点 | 错误示例 | 正确做法 | 原因 |
|--------|----------|----------|------|
| 时间片过短 | 1ms 频繁切换导致吞吐下降 | 结合切换开销设置 4~10ms | 切换成本占比过高 |
| 忽略负载类型 | 同一时间片用于 IO/CPU 混合 | IO 较小、CPU 较大分档配置 | 负载特性不同 |

## 常见错误

### ❌ 错误1: 认为时间片越短越好
**原因**：忽视切换成本，导致缓存失效与吞吐下降。  
**正确做法**：先估算一次切换成本，再让时间片至少为其数倍。

### ❌ 错误2: 用时间片处理硬实时需求
**原因**：时间片调度缺乏严格确定性。  
**正确做法**：在硬实时场景使用固定优先级抢占或 EDF。

## 实战案例

### 📚 案例: Web 服务器调度调优
**问题**：IO 密集 + 少量 CPU 密集请求，默认 4ms 时间片下高并发时吞吐下降。  
**思路**：IO 线程维持 4ms，CPU 密集线程分配 8~12ms，并对长任务启用自愿让出。  
**关键点**：分档时间片 + 自愿让出能兼顾响应与吞吐。

## 总结

### ✨ 一句话总结
时间片是抢占式调度的时间配额，把 CPU 时间量化以实现公平和可控的响应。

### 📌 核心记忆点
1. 时间片用完会触发上下文切换
2. 时间片长短决定响应与吞吐的权衡
3. 需结合负载和切换成本调优
